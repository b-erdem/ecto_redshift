# EctoRedshift

[![Hex.pm](https://img.shields.io/hexpm/v/ecto_redshift.svg)](https://hex.pm/packages/ecto_redshift)
[![Hex Docs](https://img.shields.io/badge/hex-docs-lightgreen.svg)](https://hexdocs.pm/ecto_redshift)
[![CI](https://github.com/b-erdem/ecto_redshift/actions/workflows/ci.yml/badge.svg)](https://github.com/b-erdem/ecto_redshift/actions/workflows/ci.yml)
[![License](https://img.shields.io/hexpm/l/ecto_redshift.svg)](https://github.com/b-erdem/ecto_redshift/blob/main/LICENSE)

A modern [Ecto SQL](https://hex.pm/packages/ecto_sql) adapter for
[Amazon Redshift](https://aws.amazon.com/redshift/), built on
[Postgrex](https://hex.pm/packages/postgrex).

EctoRedshift treats Redshift as a Redshift-first OLAP engine with PostgreSQL
ancestry, **not** as a drop-in PostgreSQL clone. Where Redshift diverges from
PostgreSQL (no `RETURNING`, no savepoints, no enforced constraints, no
indexes, different upsert story, `DISTKEY`/`SORTKEY`/`SUPER` and friends), the
adapter either implements the Redshift-native form or fails loudly with a
clear explanation — never silently pretending compatibility that does not
exist.

> **Why another adapter?** The previous community package,
> [`redshift_ecto`](https://hex.pm/packages/redshift_ecto), has been
> unmaintained since 2018 and targets an ancient Ecto. EctoRedshift is a fresh
> implementation against modern `ecto_sql ~> 3.13`.

## Installation

Add `:ecto_redshift` to your `mix.exs`:

```elixir
def deps do
  [
    {:ecto_redshift, "~> 0.1.0"}
  ]
end
```

Then fetch dependencies:

```shell
mix deps.get
```

## Configuration

Configure a repo the same way you would for `Ecto.Adapters.Postgres`, but
point at `Ecto.Adapters.Redshift`:

```elixir
# lib/my_app/repo.ex
defmodule MyApp.Repo do
  use Ecto.Repo,
    otp_app: :my_app,
    adapter: Ecto.Adapters.Redshift
end
```

```elixir
# config/runtime.exs
config :my_app, MyApp.Repo,
  url: System.fetch_env!("REDSHIFT_URL"),
  # e.g. "ecto://user:pass@my-cluster.abc123.us-east-1.redshift.amazonaws.com:5439/analytics"
  pool_size: 10,
  ssl: true
```

The adapter defaults the Redshift port to `5439` and uses unnamed prepared
statements, matching Redshift's preferred driver behavior.

## Quickstart: Defining a schema

Because Redshift does not support `RETURNING`, database-generated integer
primary keys are awkward with the normal `Repo.insert/2` flow. EctoRedshift
ships with a small macro that defaults schemas to application-generated
`binary_id` primary and foreign keys:

```elixir
defmodule MyApp.Analytics.Event do
  use EctoRedshift.Schema

  schema "events" do
    field :kind, :string
    field :payload, :map
    belongs_to :account, MyApp.Accounts.Account
    timestamps()
  end
end
```

Under the hood this expands to:

```elixir
@primary_key {:id, :binary_id, autogenerate: true}
@foreign_key_type :binary_id
```

You can of course opt out and manage primary keys yourself.

## Quickstart: Redshift-native migrations

EctoRedshift accepts ordinary Ecto migrations and adds first-class support
for Redshift's table options:

```elixir
defmodule MyApp.Repo.Migrations.CreateEvents do
  use Ecto.Migration

  def change do
    create table(:events,
             primary_key: false,
             options: [
               diststyle: :key,
               distkey: :account_id,
               sortkey: {:compound, [:occurred_at, :account_id]},
               backup: true,
               encode: :auto
             ]
           ) do
      add :id, :binary_id, primary_key: true
      add :account_id, :binary_id, null: false
      add :kind, :string, size: 64, null: false
      add :occurred_at, :utc_datetime, null: false
      add :payload, :super
      timestamps()
    end
  end
end
```

Supported table options: `:diststyle` (`:auto | :even | :all | :key`),
`:distkey`, `:sortkey` (atom, list, or `{:compound | :interleaved, [keys]}`),
`:backup` (`true | false | :yes | :no`), `:encode` (`:auto`).

Per-column options include `:distkey`, `:sortkey`, `:encode` (any Redshift
compression encoding: `:az64`, `:bytedict`, `:delta`, `:delta32k`, `:lzo`,
`:mostly8/16/32`, `:raw`, `:runlength`, `:text255`, `:text32k`, `:zstd`),
`:collation` (`:case_sensitive`, `:case_insensitive`, `:cs`, `:ci`, or a raw
string), plus the usual `:default`, `:null`, and `:comment`.

Identity columns are supported via `add :id, :identity, start_value: 1,
increment: 1` which emits `GENERATED BY DEFAULT AS IDENTITY(1 1)`.

## Feature support

### Supported today

- **Connection & queries** — all `Repo.all/get/one`, streaming, `explain`
- **Writes** — `Repo.insert`, `update`, `delete` with app-generated PKs
- **Bulk writes** — `insert_all` including placeholders and `INSERT INTO ... SELECT`
- **Bulk mutations** — `update_all` / `delete_all` with inner joins (Redshift `FROM`/`USING` syntax)
- **Migrations** — `CREATE` / `ALTER` / `DROP TABLE`, renames, column add/modify/remove
- **Redshift DDL** — `DISTSTYLE`, `DISTKEY`, `SORTKEY` (including compound/interleaved), `BACKUP`, `ENCODE`, `IDENTITY(seed, step)`, `:super` columns
- **Storage lifecycle** — `mix ecto.create`, `mix ecto.drop`, `mix ecto.migrate`
- **Migration locking** — via `LOCK TABLE` (Redshift has no advisory locks or DDL transactions)
- **Type mapping** — `:binary_id` / `:uuid` → `CHAR(36)`, `:map` → `VARCHAR(MAX)` with JSON round-tripping, `:super` opt-in, `:varbyte`
- **Transactions** — flat transactions via `Repo.transaction/1`

### Explicitly unsupported (fails loudly)

The adapter rejects these with a clear `ArgumentError` or `Ecto.QueryError`
rather than silently producing broken SQL:

- `RETURNING` clauses (Redshift has none)
- PostgreSQL-style `ON CONFLICT` upserts (use `MERGE` via raw SQL for now)
- Savepoints and `Ecto.Adapters.SQL.Sandbox` (Redshift has no savepoints)
- Outer-join `update_all` / `delete_all`
- `CREATE INDEX` (Redshift has no indexes — use sortkeys)
- `CHECK` / `EXCLUDE` constraints
- Array column types
- Database-generated `:id` PKs with `RETURNING`-based readback
- `ON DELETE` / `ON UPDATE` actions on foreign keys
- `structure_dump` / `structure_load`

See [`docs/compatibility.md`](docs/compatibility.md) for the authoritative list.

### Not yet wrapped (use raw SQL for now)

Redshift features that work, but don't have first-class Ecto DSL support yet
— you can always drop to `Repo.query/2`:

- `COPY` from S3 / `UNLOAD` to S3 (highest-priority roadmap item)
- `MERGE` (for upserts)
- Materialized views (`CREATE MATERIALIZED VIEW`, `REFRESH`)
- External schemas / Spectrum
- Stored procedures, `CALL`, UDFs
- `VACUUM`, `ANALYZE`, `DEEP COPY`
- `HLLSKETCH`, `GEOMETRY`, `GEOGRAPHY` types
- `SUPER` path navigation / PartiQL DSL (use `fragment(...)`)

## Testing strategy

EctoRedshift uses a two-tier live test strategy:

1. **`mix test`** — fast, pure unit tests that assert generated SQL strings.
   No database required. This is what CI runs by default.
2. **`mix test.smoke`** — driver-level smoke tests against a local PostgreSQL
   in Docker. Validates connection boot and raw SQL round-trips, but does
   **not** validate Redshift engine semantics.
3. **`mix test.integration`** — end-to-end tests against a real Amazon
   Redshift cluster or Redshift Serverless workgroup. Gated by the
   `ECTO_REDSHIFT_TEST_URL` environment variable.

```shell
# Unit tests
mix test

# Local smoke (requires Docker)
docker compose up -d postgres
ECTO_REDSHIFT_SMOKE_URL='ecto://postgres:postgres@localhost:55432/ecto_redshift_smoke' \
  mix test.smoke

# Real Redshift integration
ECTO_REDSHIFT_TEST_URL='ecto://USER:PASS@HOST:5439/DATABASE' \
  mix test.integration
```

See [`docs/testing.md`](docs/testing.md) for details.

## Contributing

Issues and pull requests are welcome. For anything beyond a typo fix or a
small bug, please open an issue first to discuss the approach — the adapter
is intentionally conservative about new features and prefers evidence from
real Redshift integration tests over theoretical compatibility claims.

When adding a feature:

1. Add unit tests that assert generated SQL strings in `test/ecto/adapters/redshift/`.
2. If the feature touches runtime behavior, add an integration test in
   `test/ecto/adapters/redshift/integration_test.exs` and verify it against
   a real cluster.
3. Update `docs/compatibility.md` and the README feature matrix.
