# Compatibility

This is the authoritative feature matrix for `ecto_redshift`. It is kept
deliberately conservative: a feature moves from "unsupported" to "supported"
only after it has unit tests asserting the generated SQL and, where
applicable, integration tests against a real Amazon Redshift cluster.

## Supported today

### Connection and queries

- `Postgrex` connection with Redshift defaults (port `5439`, unnamed
  prepared statements)
- `Repo.all`, `Repo.one`, `Repo.get`, `Repo.stream`, `Repo.explain`
- `Repo.transaction/1` (flat only — see "Unsupported today" for savepoints)

### Writes

- `Repo.insert/2` with application-generated primary keys
- `Repo.update/2`
- `Repo.delete/2`
- `Repo.insert_all/3` including placeholders
- `INSERT INTO ... SELECT` via `Repo.insert_all/3` with a subquery source
- `Repo.update_all/3` and `Repo.delete_all/2` over inner-join queries
  (Redshift `FROM` / `USING` syntax)

### Storage lifecycle

- `mix ecto.create`
- `mix ecto.drop`
- `mix ecto.migrate`
- `storage_status/1` for health checks

### Migrations

- `create`, `create_if_not_exists`, `drop`, `drop_if_exists` for tables
- `alter` with `add`, `add_if_not_exists`, `modify`, `remove`, `remove_if_exists`
- Table renames (`rename/2`) and column renames (`rename/3`)
- Foreign key references with `on_delete: :nothing` and `on_update: :nothing`
- Migration locking via `LOCK TABLE`

### Redshift DDL options

| Scope | Option | Accepted values |
|---|---|---|
| Table | `:diststyle` | `:auto`, `:even`, `:all`, `:key` |
| Table | `:distkey` | atom or string column name |
| Table | `:sortkey` | atom, string, list of columns, `{:compound, [...]}`, `{:interleaved, [...]}`, `:auto` |
| Table | `:backup` | `true`, `false`, `:yes`, `:no` |
| Table | `:encode` | `:auto` |
| Column | `:distkey` | `true`, `false` |
| Column | `:sortkey` | `true`, `false` |
| Column | `:encode` | `:az64`, `:bytedict`, `:delta`, `:delta32k`, `:lzo`, `:mostly8`, `:mostly16`, `:mostly32`, `:raw`, `:runlength`, `:text255`, `:text32k`, `:zstd` |
| Column | `:collation` | `:case_sensitive`, `:case_insensitive`, `:cs`, `:ci`, or a raw string |
| Column | `:identity` | `{seed, step}` tuple (emits `IDENTITY(seed, step)`) |
| Column | `:generated` | raw generated expression |

`ALTER COLUMN` is restricted to the safe subset Redshift actually accepts:
`VARCHAR` size changes and `ENCODE` changes. Anything else raises.

### Type mapping

| Ecto type | Redshift type |
|---|---|
| `:id` | `bigint` |
| `:identity` | `bigint GENERATED BY DEFAULT AS IDENTITY(...)` |
| `:serial` | `integer` |
| `:bigserial` | `bigint` |
| `:binary_id`, `:uuid` | `char(36)` |
| `:string` | `varchar(255)` by default; `:size` overrides; `:max` → `varchar(max)` |
| `:map`, `{:map, _}` | `varchar(max)` (JSON round-tripped) |
| `:super` | `super` |
| `:varbyte` | `varbyte(N)` |
| `:time` / `:time_usec` | `time` (with optional `:precision`) |
| `:utc_datetime` / `:utc_datetime_usec` / `:naive_datetime(_usec)` | `timestamp` (with optional `:precision`) |
| `:duration` | `interval` |
| Numeric types with `:precision` / `:scale` | Standard `type(precision, scale)` |

## Unsupported today (fail loudly)

The adapter rejects these with an `ArgumentError` or `Ecto.QueryError`
rather than silently producing broken SQL:

- **`RETURNING`** — Redshift has none.
- **`ON CONFLICT` upserts** — Redshift has no `ON CONFLICT`. Use `MERGE`
  via raw SQL until the DSL lands.
- **Savepoint-based sandbox** — Redshift has no savepoints, so
  `Ecto.Adapters.SQL.Sandbox` does not work.
- **Outer-join `update_all` / `delete_all`** — only inner joins are
  accepted.
- **`CREATE INDEX` / `DROP INDEX`** — Redshift has no indexes. Use
  sortkeys.
- **`CHECK` and `EXCLUDE` constraints** — Redshift does not enforce them.
- **Array column types** — Redshift does not support them.
- **Database-generated `:id` primary keys with `RETURNING`-based readback**
  — use `:binary_id` primary keys via `EctoRedshift.Schema` instead.
- **`ON DELETE` / `ON UPDATE` foreign key actions** — Redshift does not
  enforce referential actions.
- **`structure_dump` / `structure_load`** — returns an explicit error.
- **Constraint violation translation** — Redshift never raises them, so
  there is nothing to translate.
- **`:binary` column type** — use `:varbyte` explicitly.

## Not yet wrapped

These Redshift features work, but the adapter does not have first-class
DSL support for them. You can always drop to `Repo.query/2` with raw SQL.

- `COPY` from S3 / `UNLOAD` to S3 — the highest-priority roadmap item
- Redshift-native `MERGE` (for upserts)
- Materialized views (`CREATE MATERIALIZED VIEW`, `REFRESH`)
- External schemas / Redshift Spectrum external tables
- Stored procedures and `CALL`
- User-defined functions (SQL or Python UDFs)
- Data sharing (`CREATE DATASHARE`)
- `VACUUM`, `ANALYZE`, `DEEP COPY`
- `HLLSKETCH`, `GEOMETRY`, `GEOGRAPHY` types
- `SUPER` path navigation / PartiQL — use `fragment(...)` for now
- Late-binding views (`WITH NO SCHEMA BINDING`)
- Workload management, `SET query_group`, session variables

## Areas that still need Redshift-first decisions

Design questions that are open and will be answered by integration-test
evidence rather than speculation:

- `SUPER` read and write semantics over `Postgrex` (what does round-tripping
  nested structures actually look like at the driver level?)
- Identity columns and Ecto autogeneration behavior
- The exact shape of a Redshift-native `MERGE` DSL
- COPY/UNLOAD ergonomics (IAM role vs. credentials, manifest files,
  column mapping)
